# API 응답 상태 코드 규약

## 이 문서의 목적

`src/app/api/**/route.ts` BFF 라우트에서 어떤 상황에 어떤 HTTP 상태 코드를 반환할지 정리한다. 새 라우트를 작성하거나 리뷰할 때 참고한다. `src/server/http/response.ts`의 `jsonError`, `noContent` 헬퍼 사용을 전제로 한다.

## 규칙

| 상태 코드 | 의미 | 사용 시점 |
| --- | --- | --- |
| 200 OK | 조회/일반 성공 | 조회 또는 부수효과 없는 일반 성공 응답 |
| 201 Created | 생성 성공 | 리소스 생성 성공, 생성된 리소스를 본문으로 반환 |
| 204 No Content | 수정/삭제 성공 | 본문 없이 응답 (`noContent()`) |
| 400 Bad Request | 요청 값 검증 실패 | 필수 필드 누락, 형식 오류. 필드 자체는 유효하지만 비즈니스 규칙에 어긋나는 경우(날짜 역전 등)도 400으로 처리 |
| 401 Unauthorized | 로그인 필요 | 세션 없음. `getSessionUser` 결과가 없을 때 모든 라우트에서 공통 사용 |
| 403 Forbidden | 권한 없음 | 로그인은 됐지만 해당 리소스에 대한 권한이 없는 경우 (본인 리소스가 아닌 경우 등) |
| 404 Not Found | 리소스 없음 | 요청한 리소스가 존재하지 않는 경우. 단건 조회 시 사용 |
| 409 Conflict | 중복/상태 충돌 | 중복 생성, unique violation(23505) 등을 클라이언트에 노출해야 하는 경우 |
| 422 Unprocessable Entity | 사용하지 않음 | 400과 구분하지 않는다. 필요해지면: 400=형식/필수값 오류, 422=형식은 맞으나 비즈니스 규칙 위반 |
| 500 Internal Server Error | 예상치 못한 서버 오류 | DB 쿼리 실패 등. `cause`를 함께 넘겨 서버 로그에만 원인을 남긴다 |

## 현재 불일치 항목

- **409 미사용**: `workspaces/[id]/join`에서 unique violation을 의도적으로 무시(멱등 처리)하며, 명시적 409 응답 사례는 없다. (의도된 설계)
- **422 미사용**: 전부 400으로 통일되어 있어 문제는 없으나, 문서상 구분 기준만 명시해 둠.
